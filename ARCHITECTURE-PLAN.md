# OpenEMR AI Agent — Architecture Plan

## Overview

This project adds an AI agent layer to a fork of OpenEMR, supporting five coretools: drug interaction checks (via OpenFDA/RxNorm), symptom lookup ICD10

## Synthetic Data & Evaluation Framework

All development uses synthetic patient data generated by Synthea, producing a cohort of 150 FHIR R4 patients seeded for reproducibility and tagged by clinical profile. This dataset supports over 50 structured test cases, covering happy paths, edge cases, adversarial inputs, and multi-step reasoning scenarios.

## Core Components

The agent is built around six components:

- **Reasoning Engine:** Issues LLM calls via the Vercel AI SDK and returns structured claims with confidence scores.
- **Tool Registry:** A typed Zod-schemed catalog of all five tools, serving as the single source of truth for tool definitions and error behavior.
- **Memory System:** Uses TurboPuffer for vector-backed retrieval of conversation context and cached tool results, with Redis on Railway for short-term session state.
- **Orchestrator:** Coordinates all components via a lightweight state machine (XState or a custom graph) that manages multi-step workflows, conditional retries, and PHI boundaries.
- **Verification Layer:** Runs after every tool result and every LLM output, handling fact-checking against source APIs, hallucination detection, confidence scoring, domain constraint validation (e.g., dosage limits), and escalation for high-risk outputs.
- **Output Formatter:** Consumes verified results and renders structured UI using Vercel AI Elements, always preserving confidence indicators and source citations.

## Tech Stack

- **Frontend & LLM abstraction:** Vercel AI SDK and AI Elements
- **Model routing & cost controls:** Vercel AI Gateway
- **Vector memory:** TurboPuffer
- **Backend hosting:** Railway
- **Observability:** Helicone captures per-call latency, token costs, tool attribution, and escalation rates for an eval-driven improvement cycle.

Since all development uses Synthea-generated synthetic data, LLM access (Claude, GPT-4o) is direct via the Vercel AI Gateway and does not require HIPAA-eligible infrastructure. For real production with patient data, all LLM calls must route only through BAA-covered providers (e.g., Claude via AWS Bedrock, GPT-4o via Azure OpenAI) to meet PHI compliance — standard cloud API endpoints are not acceptable.

## Verification & Safety

The verification layer is the system’s trust backbone. Every output is assigned a confidence score. Outputs below threshold, relating to medication safety, or violating domain constraints are routed to a human-in-the-loop escalation queue rather than being shown to users. Synthea’s RxNorm-coded medication data feeds directly into the OpenFDA interaction API, ensuring drug safety checks are grounded in authoritative sources and not LLM memory.

## Data & Production Readiness

The synthetic data layer is intentionally abstracted from the orchestrator, allowing real OpenEMR data to be swapped in for production via configuration, not a rewrite. At production, PHI boundary enforcement, BAA-covered LLM hosting, and full HIPAA audit logging must be applied before any real patient data is processed.
